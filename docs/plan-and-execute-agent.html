Project Specification · v1.0

Plan-and-Execute
Agent

An agent that separates thinking from doing: a planner decomposes a goal into an ordered step sequence, an executor carries out each step with tools, and a replanner revises the remaining plan based on what actually happened.

01 · Problem Definition

What problem does this solve?

A single ReAct-style agent interleaves reasoning and action one step at a time, re-deciding the whole approach after every tool call. That's expensive, hard to inspect, and prone to losing the thread on multi-step goals. It also can't be audited — there's no artifact showing what the agent intended to do before it did it.

Problem

Complex goals need an upfront plan, not just reactive step-by-step reasoning — otherwise the agent wastes calls rediscovering structure it already knew.

Who has it

Anyone building agents for multi-step tasks: research pipelines, data workflows, ops automation — anywhere a goal decomposes into ordered, partly-dependent subtasks.

Why it matters

Plans go stale the moment reality diverges from expectation. An agent that can't revise its plan after a failed or unexpected step either stalls or barrels ahead on a broken assumption.

Expected outcome

A visible, inspectable plan; a step-by-step execution trace; and a replanner that adapts the remaining steps when execution results change what's needed.

Success definition: Given a multi-step goal (e.g., "research three competing approaches to X and produce a comparison table"), the agent produces an explicit plan, executes it with tool calls, replans when a step fails or returns unexpected results, and completes with a lower average LLM-call count than an equivalent ReAct baseline on the same task set.

02 · Goal & Scope

What are we building?

MVP (Build First)

01

Planner

LLM decomposes the input goal into an ordered list of discrete, tool-executable steps.

02

Executor

Runs one step at a time using available tools (search, code exec, file I/O), returns structured results.

03

Replanner

After each step, decides: continue with the existing plan, revise remaining steps, or declare the goal complete.

04

Step-level state tracking

Every step has a status — PENDING / RUNNING / DONE / FAILED — persisted so a run can be inspected or resumed.

05

Failure handling

A failed step triggers replanning, not silent retry — the replanner sees the failure and decides what changes.

06

Final answer synthesis

Once the plan is exhausted or the replanner marks the goal complete, results are synthesised into a final response.

07

Dynamic tool synthesis — in core v1

When executor hits a step with no matching registered tool, an LLM writes a function to solve it, runs it in the sandbox, validates output against a schema, and registers it as a reusable tool on success.

08

Hardened execution sandbox

Subprocess isolation, scratch-dir-only filesystem writes, outbound-request allowlisting, resource + time caps — load-bearing for synthesized code, not just a nice-to-have.

09

Human-in-the-loop approval gates — in core v1

Before executing a step flagged as sensitive (synthesized-tool execution, or any step touching irreversible actions like bookings/payments/sends), the graph pauses and waits for explicit approve/reject from the user before continuing.

10

Browser automation — in core v1

A stateful browser tool (navigate, click, fill, screenshot) for goals that require interacting with sites with no API — booking flows, dashboards, anything Tavily search can't reach. Separate from the synthesized-code sandbox; holds session state across multiple steps of one goal.

Out of scope (v1)

User authentication · Multi-user workspaces · Parallel step execution · Fine-tuned planner model · Long-term memory across sessions

Scope decision (updated): tool synthesis, HITL approval gates, and browser automation were all originally scoped as v2 stretch features. Moved into core v1 — this is the flagship project, no time constraint, no corner-cutting. The tradeoff is absorbed elsewhere in the master roadmap, not here.

03 · Agent Architecture

The plan → execute → replan loop

The core design choice is separating planning from execution into distinct graph nodes with their own state, rather than one node doing both — this is what makes the plan inspectable and the replanning explicit rather than implicit.

01

Receive goal & plan

User submits a goal. Planner LLM produces an ordered list of steps with a structured output schema. Steps are stored with status PENDING.

02

Execute next step

Executor takes the first PENDING step, selects and calls the appropriate tool, and captures the result (success payload or error).

03

Replan ← the adaptive step

LLM reviews the step result against the remaining plan: does the plan still make sense? If a step failed or surfaced new information, the remaining steps are rewritten.

04

No tool available? Synthesize one

Executor checks the tool registry before failing a step. No match → a synthesis prompt writes a function against a declared input/output schema, runs it in the hardened sandbox, and on a valid result registers it as a callable tool for the rest of the run.

05

Approval gate on sensitive steps

Before executing a step marked sensitive — any synthesized-tool call, or any step touching an irreversible action (booking, payment, send) — the graph pauses and surfaces the step for explicit human approve/reject. Rejection triggers a replan, not a hard stop.

06

Continue or terminate

If steps remain, loop back to execution. If the replanner determines the goal is satisfied, or a hard step-count cap is hit, the loop ends.

07

Synthesise final answer

All step results are combined into a coherent final response, with a trace of the plan actually executed (including any revisions, synthesized tools, and approval decisions).

08

Evaluate output

LLM-as-judge scores the final answer against the original goal for completeness and correctness; step-count, replan-count, synthesis-count, and approval-gate events are logged for the ablation study.

Why this beats a plain ReAct loop: ReAct re-derives structure on every turn. Plan-and-Execute derives structure once, executes cheaply against it, and only pays the expensive replanning cost when reality actually diverges — fewer LLM calls for the same task, and a plan artifact you can show in an interview.

Why tool synthesis, not just a fixed tool list: a fixed tool registry is brittle — the moment the goal needs something not pre-built (e.g. "find the cheapest flight" with no flight API wired in), a ReAct or fixed-tool Plan-and-Execute agent just fails the step. Synthesizing a tool on demand, then caching it, means the agent's capability set grows with the goals it's given instead of being capped at build time.

Why an approval gate, not full autonomy: a flagship agent that can write and run its own code is exactly the kind of system an interviewer will probe on safety. Pausing for human approval before synthesized code runs or an irreversible action fires isn't a limitation — it's the correct design for a system with this much autonomy, and it's a stronger answer than "we just trusted the sandbox."

Why browser automation, not just search: Tavily-style search returns information, not interaction — it can't select flight dates or submit a form. Any goal that requires acting on a site with no API (booking, dashboards behind a login) is unsolvable without a real browser tool. The executor treats it as a stateful tool: session persists across the multiple steps one goal needs on the same site, distinct from the disposable per-call synthesized-code sandbox.

Golden dataset (build this)

20 multi-step goals across 5 categories: (a) goals where a step fails and forces a replan, (b) goals where a step surfaces new information that changes the remaining plan, (c) straightforward goals with no replanning needed, (d) goals that require a tool not in the fixed registry — these only succeed if synthesis works, (e) goals solvable only via real browser interaction (e.g. a booking flow with no API). Human-annotate expected step count, expected final answer, and which capability (synthesis / browser / neither) is required.

Ablation study (for the blog post) — three arms

Run the same 20 goals with: (1) a plain ReAct loop with the fixed tool set, (2) Plan-and-Execute with replanning but no tool synthesis, (3) full system with synthesis. Arms 1–2 isolate the planning/replanning win (calls, wall-clock, accuracy on solvable goals). Arm 2 vs. 3 isolates the synthesis win specifically — category (d) goals should fail on arm 2 and succeed on arm 3, which is the cleanest, most defensible number in the whole study.

Hiring signal: A quantified comparison against the "obvious" baseline (ReAct), plus an isolated measurement of what the synthesis feature specifically buys you, is a stronger story than either alone — it shows you can attribute a result to a specific design decision, not just report an aggregate number.

08 · Risk Analysis

What could go wrong

High Risk

Replanner thrashing

Replanner rewrites the plan on every step even when unnecessary, erasing the efficiency win over ReAct. Mitigation: replan only triggers on FAILED status or explicit "plan invalid" signal from the LLM, not by default.

High Risk

Runaway step count

Bad plan or repeated replanning loops indefinitely. Mitigation: hard cap on total steps executed per goal, terminate with partial-answer note if hit.

Med Risk

Tool call failures

Search or code-exec tool is flaky. Mitigation: structured error capture passed to replanner as real signal, not swallowed.

Med Risk

Plan schema drift

LLM output doesn't conform to the Pydantic step schema. Mitigation: structured output mode + schema validation with one retry on parse failure.

Med Risk

Weak baseline comparison

ReAct baseline built too quickly to be a fair comparison, undermining the ablation. Mitigation: use the same tools and same LLM for both, only the orchestration differs.

Low Risk

Frontend scope creep

UI keeps expanding. Mitigation: MVP frontend is a step list with status badges only. Plan-diff visualisation is v2.

High Risk

Synthesized code is a real security surface

LLM-generated code with network access is materially different from math-only sandboxing. Mitigation: scratch-dir-only filesystem, outbound-request allowlisting, hard timeout + memory caps, no arbitrary `eval`/`exec` on unvalidated output — treat this as the project's primary security review, not a footnote.

Med Risk

Synthesis reliability

Generated tools fail in new ways — wrong auth, brittle scraping, silently wrong output that still parses. Mitigation: strict output schema validation, a synthesis-specific eval slice (category (d) goals) to catch silent failures the judge alone might miss.

Med Risk

Muddied ablation

Adding synthesis on top of replanning changes two variables at once if not isolated. Mitigation: three-arm ablation design (see Evaluation Strategy) keeps the two contributions separable.

Low Risk

Approval-gate over-triggering

Sensitivity classifier flags too many steps, making the agent feel constantly blocked on human input. Mitigation: narrow the sensitive set deliberately — synthesized-tool execution and irreversible actions only, not every tool call.

High Risk

Browser sandbox misconfiguration

agent-infra/sandbox's own quickstart runs with seccomp disabled. Mitigation: seccomp stays enabled, network restricted to an allowlist, VSCode/Jupyter components not deployed at all — only the browser piece, hardened.

Med Risk

Browser automation flakiness

Sites change layout, elements shift, timing issues. Mitigation: DOM-based selectors over brittle coordinates where possible, explicit wait conditions, treat a stuck browser step as a FAILED step the replanner can react to — not a special case.

Med Risk

Scope creep pattern

Fourth core-scope addition after synthesis, hardening, and HITL. Mitigation: any further addition gets weighed explicitly against the deadline before folding in, not added by default because the project is flagship.

09 · Security & Data

Non-functional requirements

Security — the primary review, not a footnote

All code — fixed-tool and synthesized — runs in a sandboxed subprocess with resource/time limits, never `eval`/`exec` on raw model output directly. Synthesized code specifically: scratch-dir-only filesystem writes, outbound-request allowlisting (no arbitrary domains), and output validated against a declared schema before a tool is registered for reuse. Browser automation runs in the agent-infra/sandbox browser container with seccomp enabled and network restricted to an allowlist — explicitly not the project's own quickstart defaults, which ship with seccomp disabled. All external URLs validated against a whitelist. API keys in environment variables only.

Data requirements

No training data needed. Pre-trained LLMs + live tools. The only data you build is the 20-question golden dataset, created manually over ~1 day.

Legal / privacy

No PII collected. Search results summarised, not stored verbatim. Third-party APIs used within terms of service.
