00 · End-to-End

Master execution loop

Every goal run follows the same high-level cycle. The LangGraph state machine owns transitions between nodes; SQLite persists state at each checkpoint so runs can be inspected or resumed.

Fig 0 — Full system wiring: request path, graph nodes, tool layer, and persistence
1

User submits goal

Frontend POSTs goal text to POST /goals. API creates a goal_run row (status: RUNNING) and starts the LangGraph graph.

2

Planner decomposes goal

Planner LLM returns an ordered list of steps (Pydantic schema). Stored as plan_version v1. SSE emits plan.created event to frontend.

3

Executor runs next PENDING step

Picks first PENDING step, resolves tool from registry (or routes to synthesizer). Marks step RUNNING → DONE or FAILED. SSE emits step.updated.

4

Replanner reviews result

Only triggers on FAILED status or explicit "plan invalid" signal. Rewrites remaining steps → new plan_version. SSE emits plan.revised.

5

Loop or terminate

If steps remain and cap not hit → back to Executor. If replanner marks goal complete or step cap reached → Synthesise node.

6

Final answer + eval

Combine all step results into final response. LLM-as-judge scores completeness/correctness. Persist eval_result. SSE emits run.completed.

01 · Planner

How planning works

The planner is a dedicated LangGraph node. It receives the raw goal and produces a structured, ordered step list — never executing anything itself.

Fig 1 — Planner pipeline: goal → prompt → LLM → schema validation → persisted plan
1

Build planner prompt

Inject goal text + list of available tools (search, sandbox, browser, file I/O) + output schema instructions.

2

LLM generates plan

Structured output: ordered steps, each with description, optional tool_hint, and sensitive flag for HITL.

3

Validate against PlanSchema

Pydantic enforces step count cap, non-empty descriptions, valid tool hints. One retry on parse failure.

4

Persist plan_version v1

Write to SQLite. All steps start as PENDING. Emit SSE plan.created with full step list.

5

Route to Executor

Graph edge: planner → executor (always, on first pass).

02 · Executor

How step execution works

The executor picks one step at a time, resolves the tool, runs it, and captures structured success/error output before handing off to the replanner.

Fig 2 — Executor decision tree: step selection → tool resolution → HITL check → invocation
1

Select next PENDING step

Lowest order index in current plan_version. Set status → RUNNING.

2

Resolve tool

Match tool_hint against fixed registry + dynamically synthesized tools from this run.

3

No tool found → route to Synthesizer

Conditional edge to synthesis node. On success, tool registered and step retried.

4

Sensitive step → HITL gate

If sensitive=true or synthesized-tool call: graph interrupts, waits for approve/reject.

5

Invoke tool, capture result

Structured JSON output or error object. Set step → DONE or FAILED. Persist tool_output / error.

6

Hand off to Replanner

Always routes to replanner after every step execution (replanner decides whether to actually replan).

03 · Replanner

How adaptive replanning works

The replanner is the adaptive brain. It only rewrites the plan when reality diverges — not on every successful step.

Fig 3 — Replanner decision diamond: continue, revise, or terminate
1

Receive context

Last step result (success/error), original goal, completed steps, remaining PENDING steps.

2

Check replan triggers

Replan only if: step status is FAILED, LLM signals "plan invalid", or user rejected at HITL gate.

3

Continue (no replan)

Step succeeded and plan still valid → route back to Executor for next PENDING step.

4

Revise plan

LLM rewrites remaining steps only. Persist as new plan_version. Emit SSE plan.revised with diff.

5

Goal complete

Replanner determines goal satisfied → route to Synthesise node for final answer.

6

Step cap guard

If total executed steps ≥ cap → force terminate with partial-answer note regardless of replanner opinion.

Anti-thrashing rule: Successful steps with no new information never trigger a replan. This preserves the efficiency win over ReAct.

04 · Tool Synthesis

How dynamic tool creation works

When the executor can't find a matching tool, the synthesizer generates one on demand, validates it in the sandbox, and registers it for reuse within the run.

Fig 4 — Tool synthesis lifecycle: detect gap → generate → sandbox → validate → register or fail
1

Detect missing tool

Executor finds no match in fixed registry or prior synthesized tools for this run.

2

Declare I/O schema

LLM or heuristic defines expected input params and output shape for the needed capability.

3

Generate Python function

Codegen prompt produces a function against the schema. No direct eval on raw output.

4

Execute in hardened sandbox

Subprocess with scratch-dir writes, network allowlist, time/memory caps.

5

Validate output

Result must conform to declared Pydantic schema. Silent wrong output that parses is caught by category (d) eval goals.

6

Register or fail

Success → tool keyed by task signature, added to dynamic registry. Failure → step FAILED, replanner decides next move.

7

HITL before first use

Synthesized tool execution always passes through approval gate before running on real step input.

05 · Approval Gate (HITL)

How human-in-the-loop works

LangGraph interrupt/resume pauses the graph on sensitive steps. Rejection routes to replanner, not a hard stop.

Fig 5 — HITL gate: interrupt → user decision → resume to execute or replan
1

Classify step sensitivity

Flagged at plan time (sensitive=true) or at runtime (synthesized tool, irreversible browser action).

2

Graph interrupt + checkpoint

LangGraph pauses execution. State persisted to SQLite. SSE emits gate.pending with step details.

3

User reviews in frontend

Step description, tool name, inputs shown. User clicks Approve or Reject via POST /goals/{id}/approve.

4

Approve → resume execution

Graph resumes from checkpoint. Executor proceeds with tool invocation.

5

Reject → replanner

Step marked FAILED with reason "user rejected". Replanner rewrites remaining plan.

06 · Browser Automation

How browser interaction works

Stateful browser tool for sites with no API. Session persists across steps within one goal run — distinct from the disposable code sandbox.

Fig 6 — Browser tool: executor → sandboxed container → Playwright session → structured result
1

Executor selects browser tool

Step requires site interaction (form fill, booking flow, dashboard behind login).

2

Get or create session

One browser session per goal_run. Cookies and navigation state persist across steps.

3

Execute action

navigate(url), click(selector), fill(selector, value), screenshot(). DOM selectors, not coordinates.

4

Return structured result

Page title, extracted text, screenshot path, success/error. Stuck step → FAILED → replanner.

5

HITL before irreversible actions

Final submit/booking steps trigger approval gate before click executes.

07 · Execution Sandbox

How code sandboxing works

Hardened subprocess isolation for fixed code-exec and synthesized tools. Load-bearing security boundary.

Fig 7 — Sandbox isolation: code enters boundary, validated JSON exits
Filesystem

Scratch-dir only

Writes confined to per-run temp directory. No access to host filesystem.

Network

Domain allowlist

Only pre-approved domains (e.g. api.tavily.com). Configured via OUTBOUND_DOMAIN_ALLOWLIST.

Resources

Time + memory caps

SANDBOX_TIMEOUT_SECONDS=15, SANDBOX_MAX_MEMORY_MB=256. Process killed on exceed.

Validation

Schema gate on exit

Output must parse against declared Pydantic schema before leaving sandbox or registering as tool.

08 · API + Frontend

How the UI connects to the agent

FastAPI serves REST endpoints and SSE streams. React renders live plan/step/gate updates as the graph runs.

Fig 8 — API surface and frontend components

SSE event types

plan.created

Initial plan

Full step list after planner runs. Frontend renders step cards as PENDING.

step.updated

Step status change

RUNNING → DONE/FAILED with tool output snippet.

plan.revised

Replan diff

New plan_version with changed remaining steps highlighted.

gate.pending

Approval needed

Step details for HITL. Frontend shows approve/reject buttons.

tool.synthesized

New tool registered

Tool name + schema. Shown in execution trace.

run.completed

Final answer

Goal complete with final response and eval scores.

1

User submits goal

POST /goals with { "goal": "..." }. Returns goal_run_id.

2

Frontend opens SSE stream

GET /goals/{id}/stream — receives events as graph nodes execute.

3

Live UI updates

Step badges change color: PENDING (gray) → RUNNING (blue) → DONE (green) / FAILED (red).

4

HITL interaction

On gate.pending, user clicks Approve/Reject → POST /goals/{id}/approve.

5

Run completes

run.completed event delivers final answer. UI shows full execution trace.

09 · State Persistence

How data flows to SQLite

Every graph checkpoint writes to SQLite via SQLAlchemy. Enables inspection, resume, and eval logging.

Fig 9 — Persistence layer: graph checkpoints → ORM → SQLite tables
1

goal_run created

On POST /goals: id, goal_text, status=RUNNING, created_at.

2

plan_version on plan/replan

Each planner or replanner output → new version. Steps JSON ordered list linked to version.

3

step rows updated per execution

Status transitions, tool_name, tool_input, tool_output, error persisted.

4

Checkpoint on interrupt

HITL gate saves full graph state so resume picks up exactly where it paused.

5

eval_result on completion

Judge scores, step_count, replan_count, synthesis_count, approval events logged.

10 · Evaluation

How the ablation study works

Three-arm comparison on a 20-goal golden dataset. Isolates planning efficiency gain and synthesis coverage gain separately.

Fig 10 — Three-arm ablation: same 20 goals, different orchestration, unified eval

Golden dataset categories

(a)

Forced replan

Step fails, agent must revise remaining plan.

(b)

New information

Step result changes what remaining steps should be.

(c)

Straightforward

No replanning needed. Baseline efficiency test.

(d)

Synthesis required

Tool not in fixed registry. Fails Arm 2, succeeds Arm 3.

(e)

Browser only

Requires real site interaction. Booking flows, no API.

HITL

Approval gates

Active in all arms for safety. Not a separate ablation variable.

1

Load golden goal

Each goal has expected step count, expected answer, required capability tag.

2

Run through selected arm

Same LLM, same fixed tools. Only orchestration differs between arms.

3

LLM-as-judge scores output

Completeness + correctness against original goal. Reasoning logged.

4

Log operational metrics

step_count, replan_count, synthesis_count, approval_events, browser_tool_usage.

5

Compare arms

Arm 1 vs 2 → planning efficiency. Arm 2 vs 3 → synthesis/browser coverage gain.
