00 · End-to-End

Master execution loop

Every goal run follows the same high-level cycle. The LangGraph state machine owns transitions between nodes; SQLite persists state at each checkpoint so runs can be inspected or resumed. Every node emits a LangSmith trace span (see §10 Observability) in addition to the SSE events below.

Fig 0 — Full system wiring: request path, graph nodes, tool layer, and persistence
1

User submits goal

Frontend POSTs goal text to POST /goals. API creates a goal_run row (status: RUNNING), starts a LangSmith trace for the run, and starts the LangGraph graph.

2

Planner decomposes goal

Planner LLM returns an ordered list of steps (Pydantic schema). Stored as plan_version v1. SSE emits plan.created event to frontend.

3

Executor runs next PENDING step

Picks first PENDING step, resolves tool from registry (or routes to synthesizer). Marks step RUNNING → DONE or FAILED. SSE emits step.updated.

4

Replanner reviews result

Only triggers on FAILED status or explicit "plan invalid" signal. Rewrites remaining steps → new plan_version. SSE emits plan.revised.

5

Loop or terminate

If steps remain and cap not hit → back to Executor. If replanner marks goal complete or step cap reached → Synthesise node.

6

Final answer + eval

Combine all step results into final response. LLM-as-judge scores completeness/correctness. Persist eval_result. SSE emits run.completed. LangSmith trace closed and tagged with final metrics.

01 · Planner

How planning works

The planner is a dedicated LangGraph node. It receives the raw goal and produces a structured, ordered step list — never executing anything itself.

Fig 1 — Planner pipeline: goal → prompt → LLM → schema validation → persisted plan
1

Build planner prompt

Inject goal text + list of available tools (search, sandbox, browser, file I/O) + output schema instructions.

2

LLM generates plan

Structured output: ordered steps, each with description, optional tool_hint, and sensitive flag for HITL.

3

Validate against PlanSchema

Pydantic enforces step count cap, non-empty descriptions, valid tool hints. One retry on parse failure.

4

Persist plan_version v1

Write to SQLite. All steps start as PENDING. Emit SSE plan.created with full step list.

5

Route to Executor

Graph edge: planner → executor (always, on first pass).

02 · Executor

How step execution works

The executor picks one step at a time, resolves the tool, runs it, and captures structured success/error output before handing off to the replanner.

Fig 2 — Executor decision tree: step selection → tool resolution → HITL check → invocation
1

Select next PENDING step

Lowest order index in current plan_version. Set status → RUNNING.

2

Resolve tool

Match tool_hint against fixed registry + dynamically synthesized tools (both this-run and cross-run cached tools — see §04).

3

No tool found → route to Synthesizer

Conditional edge to synthesis node. On success, tool registered and step retried.

4

Sensitive step → HITL gate

If sensitive=true or synthesized-tool call: graph interrupts, waits for approve/reject.

5

Invoke tool, capture result

Structured JSON output or error object. Set step → DONE or FAILED. Persist tool_output / error. Any content returned by a search or browser tool passes through the untrusted-content wrapper before it re-enters the LLM context (see §07 Prompt Injection Defense).

6

Hand off to Replanner

Always routes to replanner after every step execution (replanner decides whether to actually replan).

03 · Replanner

How adaptive replanning works

The replanner is the adaptive brain. It only rewrites the plan when reality diverges — not on every successful step.

Fig 3 — Replanner decision diamond: continue, revise, or terminate
1

Receive context

Last step result (success/error), original goal, completed steps, remaining PENDING steps.

2

Check replan triggers

Replan only if: step status is FAILED, LLM signals "plan invalid", user rejected at HITL gate, or a fetched/browsed content block was flagged as a possible injection attempt (see §07).

3

Continue (no replan)

Step succeeded and plan still valid → route back to Executor for next PENDING step.

4

Revise plan

LLM rewrites remaining steps only. Persist as new plan_version. Emit SSE plan.revised with diff.

5

Goal complete

Replanner determines goal satisfied → route to Synthesise node for final answer.

6

Step cap + LLM-call budget guard

If total executed steps ≥ cap, OR total LLM calls this run ≥ budget → force terminate with partial-answer note regardless of replanner opinion. The two caps are tracked separately: step cap bounds plan length, call budget bounds retries/replans/synthesis attempts within those steps.

Anti-thrashing rule: Successful steps with no new information never trigger a replan. This preserves the efficiency win over ReAct.

04 · Tool Synthesis

How dynamic tool creation works

When the executor can't find a matching tool, the synthesizer generates one on demand, validates it in the sandbox, and registers it for reuse — both within the current run and across future runs.

Fig 4 — Tool synthesis lifecycle: detect gap → check cache → generate if needed → sandbox → validate → register or fail
1

Detect missing tool

Executor finds no match in fixed registry or prior synthesized tools.

2

Check the cross-run tool cache

Tools are keyed by a normalized task signature (goal capability + I/O shape). If a prior successful synthesis already solved this signature in any earlier goal_run, reuse it directly — skip codegen and go straight to sandbox execution against the fresh step input. This is what makes the agent's capability set grow across runs, not just within one.

3

Cache miss → declare I/O schema, generate function

LLM defines expected input/output shape, then writes a Python function against it. No direct eval on raw output.

4

Execute in hardened sandbox

Same sandbox boundary regardless of cache hit or miss — even cached tools re-run inside the sandbox every time, since trusting a cached tool to run outside isolation defeats the purpose.

5

Validate output

Result must conform to declared Pydantic schema. Silent wrong output that parses is caught by category (d) eval goals.

6

Register or fail

Success → tool persisted in synthesized_tool table (durable, cross-run) and added to this run's active registry. Failure → step FAILED, replanner decides next move.

7

HITL before first use

Newly generated code always passes through the approval gate before running on real step input. A cache hit on a tool already approved in a prior run may skip re-approval — configurable, off by default for the eval build so approval events stay comparable across runs.

05 · Approval Gate (HITL)

How human-in-the-loop works

LangGraph interrupt/resume pauses the graph on sensitive steps. Rejection routes to replanner, not a hard stop.

Fig 5 — HITL gate: interrupt → user decision → resume to execute or replan
1

Classify step sensitivity

Flagged at plan time (sensitive=true) or at runtime (synthesized tool, irreversible browser action, or content flagged by the injection heuristic in §07).

2

Graph interrupt + checkpoint

LangGraph pauses execution. State persisted to SQLite. SSE emits gate.pending with step details.

3

User reviews in frontend

Step description, tool name, inputs shown. For synthesized-tool steps, the actual generated source code is rendered in full — not summarized — so the human is reviewing real code, not a paraphrase of it. User clicks Approve or Reject via POST /goals/{id}/approve.

4

Approve → resume execution

Graph resumes from checkpoint. Executor proceeds with tool invocation.

5

Reject → replanner

Step marked FAILED with reason "user rejected". Replanner rewrites remaining plan.

06 · Browser Automation

How browser interaction works

Stateful browser tool for sites with no API. Session persists across steps within one goal run — distinct from the disposable code sandbox. All content it returns is treated as untrusted (see §07).

Fig 6 — Browser tool: executor → sandboxed container → Playwright session → untrusted-content wrapper → structured result
1

Executor selects browser tool

Step requires site interaction (form fill, booking flow, dashboard behind login).

2

Get or create session

One browser session per goal_run. Cookies and navigation state persist across steps, but never leak into the planner/replanner's own context — only extracted, wrapped results cross that boundary.

3

Execute action

navigate(url), click(selector), fill(selector, value), screenshot(). DOM selectors, not coordinates.

4

Extracted text passes through injection wrapper

Page title, extracted text, screenshot path, success/error — all page-derived text is wrapped as untrusted before it re-enters any LLM context. Full detail in §07.

5

HITL before irreversible actions

Final submit/booking steps trigger approval gate before click executes.

07 · Prompt Injection Defense

How web-sourced content is kept from hijacking the agent

Anything the agent retrieves from the open web — search snippets, fetched pages, DOM text, OCR'd screenshots — is data to read, never instructions to follow. This applies uniformly to the search tool and the browser tool; both are untrusted content sources.

Fig 7 — Injection defense pipeline: wrap → scan → never auto-trigger tools from page text → flagged content forces human review
1

Delimiter + explicit framing

All fetched content — search results and browser-extracted text alike — is wrapped in a fixed template: <untrusted_web_content source="{url}">...</untrusted_web_content>. A system-level instruction accompanying every executor/replanner prompt states that content inside this tag is data to read or summarize, never a command to act on.

2

No tool calls triggered by page content alone

If page text appears to instruct a new action ("call the delete_file tool", "navigate to X and submit"), the executor cannot act on it directly. Any such suggestion must be surfaced as a proposed plan change and routed through the replanner like any other new information — never executed inline.

3

Heuristic flagging

Extracted text is scanned for injection red flags: imperative phrasing directed at an AI ("ignore previous instructions", "system:", "you must now..."), credential-looking strings, and urgency/authority language. A hit doesn't block the content from being read — it forces the current step through the HITL gate even if it wasn't originally flagged sensitive.

4

Session isolation from planning context

The browser tool's live session state (cookies, full page DOM, navigation history) never enters the planner's or replanner's context directly — only the structured, wrapped, extracted result of a single tool call does. This bounds the blast radius of a compromised page to one step's output, not the whole plan.

5

Domain allowlisting is a separate defense, not a substitute

Restricting outbound domains protects the sandbox's network egress. It does not protect against injection from a legitimate, allowlisted site whose content is attacker-controlled (a comment, forum post, or ad). Both defenses run independently — allowlisting doesn't make the content-wrapping step optional.

6

Adversarial eval coverage

The 20-goal golden dataset includes at least 1–2 goals where a fetched page contains a deliberate injection attempt, scored on whether the agent ignores it and whether the flag correctly routes to HITL.

Design principle: Trust boundaries follow data provenance, not domain reputation. A page from a trusted domain can still carry attacker-controlled content (user comments, ads, compromised third-party widgets) — so every fetched byte is untrusted by default, regardless of source, until it's been read and summarized by the LLM under the no-tool-call-from-content rule above.

08 · Execution Sandbox

How code sandboxing works

Two separate, purpose-specific sandboxes — not one shared boundary. Both are load-bearing security controls, not optional hardening.

Code-exec sandbox

Disposable, per-call subprocess

Used by the fixed code-exec tool and every synthesized-tool call (cache hit or miss — see §04). Torn down after each invocation. No state persists between calls.

Browser sandbox

Stateful, per-goal-run container

agent-infra/sandbox's browser container only — VSCode/Jupyter components excluded entirely from deployment. Session persists across steps of one goal, torn down when the run ends.

Fig 8 — Code-exec sandbox isolation: code enters boundary, validated JSON exits
Filesystem

Scratch-dir only

Writes confined to per-run temp directory. No access to host filesystem.

Network

Domain allowlist + kernel-level syscall filter

Outbound network goes through an explicit proxy with a domain allowlist (e.g. api.tavily.com), and seccomp denies raw network syscalls at the kernel level as a second layer — app-level allowlisting alone is not treated as sufficient.

Resources

Time + memory caps

SANDBOX_TIMEOUT_SECONDS=15, SANDBOX_MAX_MEMORY_MB=256. Process killed on exceed.

Validation

Schema gate on exit

Output must parse against declared Pydantic schema before leaving sandbox or registering as a (cross-run cached) tool.

Alternative considered — stated, not assumed: subprocess isolation + seccomp restricts syscalls at the app/kernel boundary but shares a kernel with the host. A microVM approach (Firecracker, gVisor) or a managed sandbox service (E2B, Modal) gives a stronger boundary — a separate kernel per execution. For this project's scope, subprocess + seccomp + allowlisting was chosen for build-time simplicity and no added infra dependency; a production deployment handling untrusted multi-tenant code would justify the migration to a microVM-based sandbox. This tradeoff is documented explicitly rather than left implicit, since it's the project's primary security review.

09 · Observability (LangSmith)

How tracing and debugging work

Every LangGraph node execution, LLM call, and tool invocation is traced via LangSmith, in addition to the SQLite persistence layer. SQLite is the system of record for plan/step state; LangSmith is the debugging and eval-authoring surface.

Run-level trace

One LangSmith trace per goal_run

Opened on POST /goals, tagged with goal_run_id, closed on run.completed with final metrics (step_count, replan_count, synthesis_count, arm) attached as trace metadata.

Node spans

Nested spans per graph node

Planner, Executor, Replanner, Synthesizer, HITL Gate, and Eval each emit a span with their prompt, raw LLM response, and parsed/validated output — this is what makes a bad replan or a bad synthesis attempt inspectable after the fact, not just logged as a pass/fail.

Tool-call spans

Every tool invocation traced

Search queries, sandbox executions, and browser actions all appear as child spans with latency and structured output — useful for spotting flaky tools (risk noted in the spec's risk analysis) directly in the trace rather than inferring it from step failures.

Eval dataset tooling

Overlaps with the custom LLM-judge harness

LangSmith's dataset/eval features can host the 20-goal golden dataset and run the three-arm comparison natively. Decision: use LangSmith for trace inspection and dataset storage; keep the custom LLM-as-judge scorer (Section 07 of the spec) rather than LangSmith's built-in evaluators, since the judge rubric (completeness/correctness against a human-annotated expected answer) is project-specific.

Why this matters for the resume story: a trace viewer showing exactly which LLM call caused a bad replan, or why a synthesized tool's output failed schema validation, is a much stronger interview artifact than a log line — it's the difference between "the agent sometimes fails" and "here's the exact prompt and response that caused this specific failure mode."

11 · State Persistence

How data flows to SQLite

Every graph checkpoint writes to SQLite via SQLAlchemy. Enables inspection, resume, and eval logging. LangSmith traces are the parallel, richer debugging layer (§09) — SQLite remains the source of truth for plan/step state and the cross-run tool cache.

Fig 10 — Persistence layer: graph checkpoints → ORM → SQLite tables
1

goal_run created

On POST /goals: id, goal_text, status=RUNNING, created_at, langsmith_trace_id.

2

plan_version on plan/replan

Each planner or replanner output → new version. Steps JSON ordered list linked to version.

3

step rows updated per execution

Status transitions, tool_name, tool_input, tool_output, error persisted.

4

synthesized_tool is cross-run durable

Unlike other tables scoped to one goal_run, this table persists across runs — keyed by task signature, it's the cache checked in §04 step 2.

5

Checkpoint on interrupt

HITL gate saves full graph state so resume picks up exactly where it paused.

6

eval_result on completion

Judge scores, step_count, replan_count, synthesis_count, approval events, injection flags logged.
