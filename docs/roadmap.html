Changelog · v1 → v2

What changed and why

1. Arm labels corrected

v1 called the planner-only phase "Arm 1" and the executor phase "Arm 2" — but a planner-only system isn't a ReAct baseline, and no phase actually built one. Per the spec's ablation design: Arm 1 = plain ReAct (fixed tools, no explicit plan — now built explicitly in Phase 8), Arm 2 = Plan-and-Execute + replanning, no synthesis (complete at end of Phase 3), Arm 3 = full system + tool synthesis + browser (complete at end of Phase 7).

2. "Synthesis" redefined to match the spec

v1 Phase 4 described "synthesis" as generating multiple alternative plans and merging them — a different feature from what the spec and system-wiring actually define. The spec's synthesis is dynamic tool synthesis: when the executor hits a step with no matching tool, an LLM writes a function against a declared schema, runs it in the sandbox, and registers it as a reusable tool. Phase 4 below reflects that.

3. HITL approval gates and browser automation added as core phases

v1 had a generic "safety layer" (Phase 5) and browser tool (Phase 6) that didn't match the spec's specific HITL interrupt/resume design or the seccomp-hardened browser sandbox. These are now Phase 6 and Phase 7, matching system-wiring's Fig 5 and Fig 6 exactly.

4. Timeline aligned to spec dates

Phases now map 1:1 to the spec's Section 06 timeline (Jul 7 – Aug 10, 35 days). This roadmap is the phase-by-phase execution view of that timeline, not a separate plan.

5. New: gap fixes folded into relevant phases

Phase 4 adds an LLM-call budget cap and cross-run tool-registry caching (previously unaddressed — tools were only cached within a single run). Phase 5 adds an explicit sandbox-alternative decision note (subprocess+seccomp vs. microVM/managed sandbox). Phase 7 adds a prompt-injection defense for browser-sourced content, and its HITL gate now surfaces generated source code for review, not just the step description. Phase 0 adds a lightweight testing/CI task that v1 omitted entirely.

Phase 4 · Jul 25–27
4

Dynamic Tool Synthesis

3 days · CLI only
Building toward Arm 3
Pending

Tasks

  • Missing-tool detection in executor (no registry match)
  • Codegen prompt: LLM writes a Python function against a declared I/O schema
  • Run generated code in the (unhardened-for-now) sandbox, validate output
  • On success, register as a reusable tool keyed by task signature
  • Cross-run tool registry — persist successful synthesized tools beyond a single goal_run so later goals reuse them instead of re-synthesizing
  • LLM-call budget cap per goal_run, separate from the step-count cap, so a runaway replan/synthesis-retry loop can't burn unbounded calls before the step cap catches it

Deliverables

  • Synthesizer node with declared input/output schema
  • Pydantic output validation gate
  • Persistent, versioned synthesized-tool registry (cross-run)
  • LLM-call counter + configurable budget cap

You can: Give the agent a goal that needs a tool it doesn't have. It writes one, validates it, and uses it — and if the same capability gap shows up in a later goal, it reuses the cached tool instead of synthesizing again.

Phase 7 · Aug 1–4

Browser Automation

4 days · CLI only
Arm 3 complete at end of this phase
Pending

Tasks

  • agent-infra/sandbox browser container — seccomp enabled, network allowlisted (not the project's quickstart defaults)
  • navigate / click / fill / screenshot tools, DOM selectors not coordinates
  • Session persistence across steps within one goal_run
  • Stuck browser step → FAILED → replanner, not a special case
  • HITL gate before irreversible actions (final submit/booking click)
  • Prompt-injection defense: page content/DOM text extracted by the browser tool is treated as untrusted data, not instructions — wrap extracted text so the executor LLM cannot interpret in-page text as new commands, and flag pages containing suspicious imperative-looking text for the HITL gate

Deliverables

  • Browser tool set wired into the registry
  • Hardened container config (VSCode/Jupyter components excluded)
  • Browser session-state persistence
  • Injection-resistant content extraction wrapper

You can: Give the agent goals that require real site interaction with no API — booking flows, dashboards behind a login. This completes Arm 3 — the full system (planning + replanning + tool synthesis + browser), ready for the ablation in Phase 8.
